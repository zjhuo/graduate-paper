# 真实 PUF 前端接入实施清单与验收条件

日期：2026-06-03

## 1. 文档目的

本文档用于把前面已经整理好的：

- 前端接入方案；
- wrapper 接口与时序 spec；
- 采样字段与指标口径 spec；
- 采样实验计划与数据流说明；

进一步收敛成一份可执行的实施清单。

本文档只做任务拆解与验收条件定义，不修改 RTL，不运行 Vivado，不声明真实前端已经接入完成。

---

## 2. 当前已具备的前置条件

当前已经具备：

1. 基础 64 级 APUF RTL 原型；
2. 固定/模拟响应源条件下的后处理与认证主链闭环；
3. `response_aggregate_ctrl`、Hamming FE、SPONGENT/KDF、`auth_pass` 已硬件化；
4. 真实前端 wrapper 的目标接口与时序 spec；
5. 真实采样字段、指标口径和实验计划文档。

当前尚未具备：

1. 已接入主链的真实 APUF 前端；
2. 真实前端下的 `puf_resp_valid` 正式握手；
3. challenge settle / launch / sample 参数实测值；
4. 真实采样日志；
5. 真实多板、多温、重复采样统计结果。

---

## 3. 后续实施总目标

后续实施的最小目标是：

> 用真实 PUF 前端 wrapper 替换当前模拟响应源，同时保持后级 `response_aggregate_ctrl`、Hamming FE、SPONGENT/KDF、protocol FSM 和 `auth_pass` 主链不变，并在真实采样条件下重新验证整条链路。

---

## 4. 分阶段实施清单

建议分为 P0、P1、P2 三个层级推进。

---

## 5. P0：必须完成

### P0-1 前端 wrapper RTL 接口落地

#### 目标
把 `real_puf_frontend_wrapper` 按既定 spec 实现为可插入的前端适配层。

#### 输入
- 基础 APUF RTL 原型；
- `real_puf_frontend_wrapper` 接口与时序 spec；
- 当前 `board_bringup_top` 与 `apuf_capture_ctrl` 边界。

#### 输出
- `real_puf_frontend_wrapper` RTL；
- 接口包括：
  - `clk`
  - `rst_n`
  - `sample_req_i`
  - `challenge_i[63:0]`
  - `challenge_valid_i`
  - `puf_resp_o`
  - `puf_resp_valid_o`
  - `busy_o`（可选）
  - `timeout_o`（可选）

#### 验收条件
- 不改后级主链接口；
- `sample_req_i` 与 `puf_resp_valid_o` 能建立一一对应关系；
- challenge 锁存与稳定等待逻辑明确存在；
- `launch` 与 sample 时序明确存在。

#### 风险
- `puf_resp_valid_o` 生成不稳定；
- challenge 未稳定即采样；
- 参数过于拍脑袋，没有板级依据。

---

### P0-2 用真实前端替换模拟响应源

#### 目标
将当前 `board_bringup_top` 中的固定/模拟响应源替换为真实前端 wrapper。

#### 输入
- 已实现的 `real_puf_frontend_wrapper`；
- 当前 bring-up top。

#### 输出
- 新版 bring-up top（或等效接入版本）；
- `puf_resp` 不再来自内部计数器/固定响应，而来自真实前端 wrapper。

#### 验收条件
- 后级 `response_aggregate_ctrl`、Hamming FE、SPONGENT/KDF、`auth_pass` 不改接口；
- 模拟响应源逻辑被替换掉；
- 系统仍能形成 `sample_req -> puf_resp_valid -> aggregate -> FE -> auth_pass` 的完整链。

#### 风险
- 前端接入后，后级大量失败，导致难以区分是前端问题还是后级问题；
- 因此必须保留中间观测口径和离线日志。

---

### P0-3 单板单温连通性验证

#### 目标
先证明真实前端在最小条件下可采样。

#### 输入
- 1 块板；
- 室温；
- 少量 challenge；
- 重复采样设置。

#### 输出
- 原始采样日志；
- `sample_req` 与 `puf_resp_valid` 对应关系记录；
- 简单稳定性观察结果。

#### 验收条件
- 每次 `sample_req` 能得到对应响应或明确 timeout；
- `puf_resp_valid` 不是乱跳；
- raw response 不是全常量或完全不可重复。

#### 风险
- APUF 前端根本不起作用；
- 时序参数完全不合适；
- 响应信号过于不稳定，无法进入后级。

---

### P0-4 单板完整 challenge 子表稳定性验证

#### 目标
在一块板上确认当前 challenge 子表在真实前端下是否足够稳定。

#### 输入
- 1 块板；
- 注册冻结的受限 challenge 子表；
- 多次重复采样。

#### 输出
- raw BER；
- reliability；
- aggregate BER；
- FE recovery success rate；
- 单板 `auth_pass` 成功率。

#### 验收条件
- 聚合后误码率明显低于原始误码率；
- FE 能在可接受比例上恢复成功；
- `auth_pass` 至少在单板固定条件下可重复成功。

#### 风险
- 当前 challenge 子表在真实前端下不够稳；
- `helper_xor / checksum / s_tag` 需要重新生成；
- 现有 `N_REP=5` 可能不够。

---

## 6. P1：强烈建议完成

### P1-1 多板共同 challenge set 采样

#### 目标
获得有限样本 FPGA 级的 inter-chip 差异指标。

#### 输入
- 3~5 块板；
- 共同 challenge set；
- 每板多次重复采样。

#### 输出
- uniformity；
- uniqueness；
- reliability；
- bit-aliasing。

#### 验收条件
- uniqueness 严格基于共同 challenge set 统计；
- 不混入 per-device BestIdx 口径；
- 每个指标都有原始数据可回溯。

#### 风险
- 板数有限，只能支撑有限样本结论；
- 指标容易被误写成“大规模芯片结论”。

---

### P1-2 多温扩展验证

#### 目标
评估温度变化对真实前端与认证链的影响。

#### 输入
- 3~5 块板；
- 室温；
- 条件允许时加入低温 / 高温。

#### 输出
- 温度条件下的 raw BER / aggregate BER；
- FE recovery success rate；
- auth success rate。

#### 验收条件
- 温度标签记录完整；
- 同一 challenge set 下可比较；
- 能给出是否需要从 `N_REP=5` 升到 `N_REP=7` 的判断依据。

#### 风险
- 板级条件不好控制；
- 数据量不足以得出太强结论。

---

### P1-3 常量重新生成与重绑定

#### 目标
在真实前端下重建与当前设备匹配的常量。

#### 输入
- 真实采样数据；
- challenge 子表；
- 当前 helper / checksum / `s_tag` 定义。

#### 输出
- 新的 `helper_xor`
- 新的 `helper_mask`
- 新的 `checksum`
- 新的 `s_tag` 或协议相关派生值

#### 验收条件
- 所有常量版本可追踪；
- 文档中能明确区分固定向量版本与真实前端版本；
- `auth_pass` 基于真实响应重新验证。

#### 风险
- 当前固定向量版本和真实版本混写；
- 常量变更后旧回归结果失效但未重新记录。

---

## 7. P2：可作为扩展

### P2-1 动态常量加载
目标：从 wrapper 固定常量推进到 PS/UART/AXI-Lite 动态加载。

### P2-2 PS 协同软件驱动
目标：补 start 写入、状态读取、常量装载、结果检查的软件路径。

### P2-3 完整版 ProVerif v2 收敛
目标：在状态推进基础上进一步增强形式化模型。

### P2-4 NIST 与建模攻击评估
目标：补随机性统计和必要的建模攻击边界说明。

---

## 8. 后续每一步建议保留的证据

后续每个阶段至少应保留以下证据之一：

- 原始采样日志；
- challenge 级聚合统计表；
- FE / `auth_pass` 成功率汇总表；
- 关键参数说明；
- 板号、温度、版本说明；
- 失败案例记录。

这样后续无论写论文、做汇报还是回看问题，都能明确：

- 哪一步已经完成；
- 哪一步只是计划；
- 哪一步失败过以及为什么失败。

---

## 9. 不应提前宣称的内容

在真实前端未完成上述 P0 验收前，不应写成：

- 真实 APUF 前端已完成；
- 真实 APUF response 已跑通 `auth_pass`；
- 真实 PUF 指标已最终确认；
- 多板 uniqueness 已得到最终结论；
- 当前固定 challenge 子表已经在真实前端下完全验证。

---

## 10. 当前阶段结论

当前最合理的推进方式是：

1. 先把真实前端 wrapper 接口与时序落地；
2. 再替换模拟响应源；
3. 再做单板连通性与稳定性验证；
4. 最后逐步扩展到多板、多温、真实认证成功率统计。

因此，当前最准确的结论是：

> 真实 PUF 前端接入的实施路径已经能够拆解成明确的 P0/P1/P2 任务，但实际接入、实测和统计仍未开始。
