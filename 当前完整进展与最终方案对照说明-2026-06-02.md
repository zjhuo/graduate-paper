## 1. 最终完整方案是什么

### 1.1 方案定位

本方案**不是**单纯提出一个新的 APUF 单元电路，也**不是**单纯提出一个软件认证协议。更准确的定位是：

> **面向 IoT 终端的 PUF 稳定响应生成与轻量 AKA 硬件闭环方案。**

其中，“硬件闭环”强调的是：从 challenge 读取、重复采样、响应聚合、模糊提取、密钥派生、标签计算、认证通过判决，到板级 bring-up 入口和形式化验证，已经形成一条可以继续落地的工程链路。

### 1.2 完整目标系统的结构和数据链路

最终完整方案的目标链路如下：

1. **真实 PUF 前端**接收 challenge 并输出原始响应；
2. challenge 来自**注册冻结的受限挑战子表**（也可表述为“离线筛选得到的稳定挑战子表”）；
3. 每个 challenge 默认重复采样 `N_REP=5` 次；
4. `response_aggregate_ctrl` 对 5 次响应做 majority（多数表决），并产生 unreliable bit（不可靠位）标记；
5. 聚合响应与 `helper_xor / helper_mask` 进入 Hamming FE（模糊提取器），恢复 `message_bits_hat`；
6. `message_bits_hat` 进入 SPONGENT-KDF 路径，计算：
   - `checksum_calc`
   - `K_hat`
   - `H_tag`
   - `S_tag` 验证相关派生量；
7. 协议层按四报文交互：
   - `M1 = DeviceID, nonce_d`
   - `M2 = nonce_s, V_i, C_init`
   - `M3 = nonce_d, H_tag`
   - `M4 = S_tag`
8. 设备端根据 `checksum_match` 与 `S_tag` 验证结果给出 `auth_pass`；
9. 服务端在认证成功后推进状态（例如 `Seed_i -> Seed_next`），失败则不更新状态；
10. 常量加载从 v1 固定向量 bring-up 逐步演进到 v2 半动态实验，再演进到 v3 PS/UART/AXI-Lite 动态加载；
11. 板级接口只保留正式最小口，调试口不再作为高速外部同步接口；
12. 安全性由 ProVerif 最小模型与状态推进模型提供协议级支撑。

### 1.3 当前工程中的“最终方案”和“当前基线”的关系

需要明确区分：

- **最终完整方案**：包含真实 PUF 前端、真实采样时序、动态常量加载、状态推进、系统化接口、真实板级实验和更完整的形式化验证。
- **当前硬件主基线**：已经完成主链功能闭环、协议语义回归、board bring-up 入口、Z7 Lite 板卡约束、bitstream 生成，以及 ProVerif v1 / v2-lite 支撑，但当前上板入口仍采用**固定/模拟响应源**，真实 PUF 前端尚未接入。

因此，当前版本已经不是“只有仿真想法”的阶段，但也还不是“真实 PUF 与动态系统全部落地”的最终版本。

### 1.4 方案定位与创新点表达

建议后续论文或汇报中，把创新点分成两类来讲，而不要把本方案表述成“提出了一种全新的 PUF 单元结构”。

#### A. 面向认证链路的 PUF 稳定响应轻量生成

- 离线筛选 challenge；
- 使用注册冻结的受限挑战子表；
- 默认 `N_REP=5` 重复采样；
- majority 聚合与 unreliable bit 协同；
- Hamming FE 与 helper data 协同恢复稳定密钥材料。

#### B. 无在线 CRP 暴露的 PUF-AKA 硬件闭环

- 不在线传输原始 response；
- 使用 `K_hat`、SPONGENT-KDF、`H_tag`、`S_tag`、`auth_pass` 构成轻量认证链；
- 板级 bring-up top 已形成最小可下载入口；
- ProVerif v1 支撑最小双向认证与会话密钥保密；
- ProVerif v2-lite 支撑“成功才推进状态、失败不更新状态”。

### 1.6 固定/受限挑战子表的安全边界说明

后续文档中不建议简单写“固定 challenge”，而建议统一表述为：

- **注册冻结的受限挑战子表**；或
- **离线筛选得到的稳定挑战子表**。

其安全边界应明确为：

1. challenge 子表**不作为长期秘密**；
2. 安全目标**不是隐藏 challenge 本身**，而是避免在线协议暴露原始 CRP；
3. 在线协议当前只传输 `DeviceID`、nonce、`H_tag`、`S_tag` 以及相关派生量，不传原始 response；
4. helper data 视为公开辅助数据，但攻击者不应只靠 helper 推出长期密钥材料；
5. ProVerif v1 当前证明的是：
   - `SK secrecy`
   - Server authenticates Device
   - Device authenticates Server
6. ProVerif v2-lite 当前证明的是：
   - 成功才推进状态；
   - 失败不更新状态。

同时也必须明确：

- ProVerif **不证明** PUF 抗机器学习建模能力；
- ProVerif **不证明** 真实熵；
- ProVerif **不证明** NIST 随机性；
- 这些内容需要由真实 PUF 指标、NIST SP 800-22 随机性测试，以及必要的建模攻击评估来补充。

### 2.1 `recover_success_o` 的当前语义问题

当前 bring-up 场景中，`recover_success_o` 的观测结果与 `auth_pass` 关系并不清楚。已有 bring-up 日志显示：

- 正确路径下 `auth_pass=1`，但 `recover_success_o` 仍可能为 0；
- 因此它**不适合作为当前正式成功判据**。

当前更稳妥的结论是：

- **正式成功判据以 `auth_pass` 为主**；
- `recover_success_o` 更像某个中间阶段或内部状态；
- 后续如果继续保留它，需明确它到底表示：
  - FE 成功；
  - checksum 成功；
  - 还是某个中间恢复阶段已完成。

如果后续仍语义不清，建议：

- 重命名；
- 重新定义；或
- 从正式板级口中移除。

### 2.2 `per-device BestIdx` 与固定 challenge ROM 的关系

当前文档中必须避免把“每设备 BestIdx”和“全局固定 challenge ROM”写成互相冲突的两个版本。更准确的表述应是：

- **v1**：固定 ROM 服务固定向量 bring-up；
- **v2**：支持多设备替换 `BestIdx / challenge ROM` 的半动态实验；
- **v3**：由注册阶段为每个设备生成独立挑战子表，并通过文件、BRAM、PS、UART 或 AXI-Lite 方式加载。

也就是说，当前 `fixed_challenge_table` 与 `fixed_challenges_256x64.hex` 代表的是 **v1 工程基线**，并不否定后续演进到每设备独立 challenge 子表。

### 3.1 SDK / 软件驱动测试的定位

当前 v1 是 **PL-only 固定向量 bring-up**，不一定需要 SDK 驱动；当前 bring-up top 把测试握手收回板内，已经可以在 PL 侧形成固定向量闭环。

但如果后续进入 **PS/UART/AXI-Lite 动态加载版本**，则需要补：

- Xilinx SDK / bare-metal 软件驱动；
- `start` 写入；
- 状态读取；
- 常量加载；
- `auth_pass` 检查；
- 必要时的错误路径触发与日志输出。

因此，SDK 驱动不是当前主链 bring-up 的前提条件，但属于后续系统化版本的重要待完成项。

### 3.2 真实 PUF 多板多温采样计划

当前尚未做真实 PUF 采样，但后续应至少规划：

- **3–5 块板**；
- 至少覆盖**室温**；条件允许时加入**低温 / 高温**；
- 每块板采集多个 challenge；
- 每个 challenge 做多次重复采样；
- 统计指标包括：
  - raw BER
  - aggregate BER
  - uniformity
  - uniqueness
  - reliability
  - bit-aliasing
  - FE recovery success rate
  - auth success rate

需要明确：

- 这类实验属于**有限样本 FPGA 级验证**；
- 不能夸大为大规模芯片统计结论；
- 真实结果不能预先断言“更好”或“更差”，只能实测后再下结论。

### 3.3 NIST SP 800-22 随机性测试计划

NIST SP 800-22 可作为后续补充指标，用于对以下序列做统计随机性检查：

- raw response 序列；
- 聚合后响应序列；
- KDF 输出序列。

但需要明确：

- NIST 仅是**统计随机性补充评估**；
- 它**不能替代** PUF 的 uniformity、uniqueness、reliability、BER、bit-aliasing；
- 它也**不能等同于安全性证明**；
- 如果当前数据长度不足，应明确写为“计划项/补充项”，不能写成“已完成”。

#### 4.7.2 `uniqueness = 56.7104%` 的风险说明

这个结果**不能当作最终理想 PUF 唯一性指标**，原因包括：

1. 它来自 **Python / 行为仿真中间结果**，不是实时板级 PUF 实测；
2. 它必须进一步检查：
   - 是否是在**共同 challenge set** 上计算；
   - 是否存在“筛选过程改变传统 uniqueness 含义”的问题；
3. 当前 `baseline_bestidx_balanced` 使用的是一个全局筛选后的稳定 challenge 子集，因此它更适合作为：
   - 当前离线筛选策略下的**中间统计结果**；
   - 而不是直接替代传统 inter-chip uniqueness 结论；
4. 如果后续演进到 **per-device BestIdx / per-device challenge ROM**，不同设备使用不同 challenge 子表，则该指标更不能直接称作传统 inter-chip uniqueness，更合适的表述是：
   - **selected response diversity（筛选后响应材料差异）**。

因此，当前**不能**用 `56.7104%` 作为最终 PUF 唯一性结论。

#### 4.7.3 PUF 指标异常与后续修正计划

后续需要明确补做：

- 重新检查 uniqueness 的统计口径；
- 补充 bit-aliasing；
- 补充 NIST SP 800-22 随机性测试；
- 在真实 3–5 块板、多温度、多次采样下重新统计：
  - uniformity
  - uniqueness
  - reliability
  - BER
  - bit-aliasing
  - FE recovery success rate
  - auth success rate。

需要再次强调：

- 3–5 块板的实验，只能支撑**有限样本 FPGA 级验证**；
- 不能夸大为大规模芯片统计结论；
- 真实 PUF 指标必须和 Python/行为仿真指标分开写。

#### 4.7.4 NIST SP 800-22 随机性测试计划

NIST SP 800-22 将作为后续补充指标，用于评估：

- raw response 序列；
- 聚合响应序列；
- KDF 输出序列；
- 必要时还可扩展到标签序列。

但它的定位必须克制：

- NIST 只评估**统计随机性**；
- 不能替代 PUF 的 uniformity、uniqueness、reliability、BER、bit-aliasing；
- 不能替代真实安全性证明；
- 更不能替代 ProVerif 或真实前端实测。

当前由于真实板级数据量不足，NIST 应写成**补充项 / 计划项**，而不是已完成。

## 7. 当前最准确的阶段结论

截至目前，当前方案已经达到：

- **硬件主链功能闭环**：从受限挑战子表、重复采样、聚合、Hamming FE、SPONGENT/KDF 到 `auth_pass` 的主链已打通；
- **验证支撑较完整**：标准 tb、regression、bring-up 行为回归、ProVerif v1、ProVerif v2-lite 都已形成证据；
- **工程化程度高于纯仿真方案**：Z7 Lite bring-up top、板卡专用约束和 bitstream 已完成，50MHz post-route timing 已通过；
- **当前可作为上板前硬件主基线**。

但当前**仍未完成**：

- 真实 PUF 前端接入；
- 真实采样数据下 `helper / checksum / s_tag` 的重新生成与验证；
- 动态常量加载；
- 完整 PS 系统化版本；
- 完整版 v2 状态模型验证；
- 真实上板实验记录；
- NIST、bit-aliasing、真实多板多温 PUF 指标统计。

因此，当前最准确的判断不是“方案全部完成”，而是：

> **硬件主链、协议语义和 bring-up 工程入口已经站住；当前阶段的重点不再是“主链能不能跑”，而是把真实 PUF 前端、多板实测、动态加载和系统化验证补齐。**

## 9. 简短结论（适合组会/论文摘要口径）

当前项目已经形成了一版较完整的上板前硬件主基线：

- 硬件主链从注册冻结的受限挑战子表、重复采样、聚合、Hamming FE、SPONGENT/KDF 到 `auth_pass` 已闭环；
- 标准仿真、错误路径回归、bring-up 行为回归都已通过；
- ProVerif v1 已验证最小双向认证和会话密钥保密性，v2-lite 已验证“成功才推进状态、失败不更新状态”；
- Z7 Lite bring-up top 的 50MHz synthesis、post-route 和 bitstream 已完成。

但当前版本仍然是“上板前工程主基线”，不是完整终版：

- 真实 PUF 前端尚未接入；
- 动态常量加载尚未实现；
- 完整版 v2 状态模型尚未收敛；
- 真实板级记录、真实 PUF 指标、NIST 和多板多温实验尚未补齐。

因此，当前最合理的结论是：

> **项目已经从“只有设计想法”推进到“硬件主链、协议语义和板级 bring-up 入口都已站住”的阶段；后续工作的重点，将从主链搭建转向真实前端接入、真实实验补证和系统化扩展。**
