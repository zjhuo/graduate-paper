# 两次上板计划对比说明（2026-06-04）

## 1. 目的
本文档用于区分项目中的两次“上板”准备，它们目标不同、输入不同、验证重点也不同。若不提前区分，后续论文、汇报或开发推进时，容易把“板级 bring-up 已完成”和“真实 PUF 已完成验证”混写。

## 2. 第一次上板：固定/模拟响应源 bring-up

### 2.1 目标
第一次上板的目标是：
- 先验证板级工程能跑起来；
- 先证明 `start -> busy -> done -> auth_pass` 这条硬件主链闭环成立；
- 先把时钟、复位、按键、LED、约束、bitstream、post-route 时序这些工程问题解决掉。

### 2.2 前提
第一次上板默认：
- 不接真实 PUF 前端；
- 不依赖真实物理延迟竞争；
- 使用板内固定/模拟响应源；
- 使用固定版本的 `helper_xor`、`checksum`、`s_tag` 等常量。

### 2.3 主要验证内容
- 板级接口是否正确；
- bring-up top 是否可综合、可实现、可生成 bitstream；
- `auth_done`、`checksum_match`、`auth_pass` 在固定向量下是否符合预期；
- 板级时钟约束和输出口时序是否可收敛。

### 2.4 产出
第一次上板已经形成：
- `iotpufs_terminal_board_bringup_top.sv`
- Z7 Lite 约束与 bitstream
- bring-up checklist
- 板级接口与常量加载清单
- post-route 通过的板级入口

### 2.5 能说明什么
第一次上板能说明：
- 后级硬件主链已闭环；
- 板级 bring-up 路径可行；
- 固定/模拟响应源条件下认证链能通过。

### 2.6 不能说明什么
第一次上板不能说明：
- 真实 PUF 前端已接入；
- 真实 PUF response 已通过认证；
- 真实 BER、reliability、uniqueness 等已得到。

## 3. 第二次上板：真实 PUF 前端接入与认证重绑定

### 3.1 目标
第二次上板的目标是：
- 接入真实 PUF 前端；
- 获取真实板级采样数据；
- 基于真实采样结果重新生成 `helper_xor`、`checksum`、`s_tag` 等常量；
- 在真实前端条件下恢复 `auth_pass` 闭环；
- 得到真实 PUF 指标和系统链路指标。

### 3.2 前提
第二次上板需要：
- `real_puf_frontend_wrapper` 或等价真实前端接入层；
- 真实 challenge -> response 采样链；
- challenge settle、launch、sample、valid 规则；
- 真实采样日志和离线统计流程。

### 3.3 主要验证内容
- 真实前端是否能输出有效 `puf_resp_valid`；
- `N_REP=5` 聚合后 BER 是否可接受；
- FE 是否能稳定恢复；
- 新生成的常量是否能使 `checksum_match`、`auth_pass` 重新成立；
- 真实 PUF 的 BER、reliability、uniformity、uniqueness、bit-aliasing 是否达到可接受水平。

### 3.4 产出
第二次上板预期产出：
- 真实采样日志；
- 聚合与认证汇总；
- 真实前端条件下的 `helper_xor`、`checksum`、`s_tag` 等新常量；
- 真实 PUF 指标；
- 真实认证成功率。

### 3.5 能说明什么
第二次上板如果完成，才能说明：
- 真实前端已经接入；
- 真实 PUF response 能驱动后级 FE/KDF/auth_pass 闭环；
- 真实 PUF 指标和真实认证成功率已得到实测证据。

## 4. 两次上板的核心差别
| 对比项 | 第一次上板 | 第二次上板 |
|---|---|---|
| 主要目的 | 先把板级工程跑起来 | 用真实 PUF 数据恢复认证闭环 |
| 前端来源 | 固定/模拟响应源 | 真实 PUF 前端 |
| 常量来源 | 当前 wrapper 固定常量 | 真实采样后重新生成并重绑定 |
| 核心验证 | 主链闭环、时序、bitstream | 真实采样、真实 BER、FE 恢复、真实 auth_pass |
| 能否证明真实 PUF 已验证 | 不能 | 可以（前提是完成实测） |
| 当前状态 | 已形成入口并具备 bitstream | 仍在准备和接口定义阶段 |

## 5. 当前阶段最准确的说法
当前项目已经完成的是：
- 第一次上板所需的主要工程准备；
- 固定/模拟响应源条件下的板级 bring-up 闭环。

当前尚未完成的是：
- 第二次上板，即真实 PUF 前端接入、真实采样、常量重生成与认证重绑定。

## 6. 简短结论
如果把两次上板混成一件事，最容易造成的误解就是：把“固定/模拟响应源下主链已跑通”误写成“真实 PUF 已验证通过”。因此，后续文档、汇报和论文中应明确区分：第一次上板解决的是工程 bring-up，第二次上板解决的才是真实前端、真实采样和真实认证闭环。
