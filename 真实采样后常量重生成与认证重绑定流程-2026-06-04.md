# 真实采样后常量重生成与认证重绑定流程（2026-06-04）

## 1. 目的
本文档用于说明：在后续接入真实 PUF 前端并取得真实采样数据之后，如何重新生成 `helper_xor`、`helper_mask`、`checksum`、`s_tag` 等常量，并将其重新绑定到当前硬件主链，使系统从“固定/模拟响应源闭环”过渡到“真实采样数据闭环”。

本文档是流程和接口说明，不等于真实实验结果。当前尚未完成真实前端上板采样，因此本文档不能作为“真实 PUF 已验证通过”的证据。

## 2. 适用范围
适用于后续以下场景：
- 使用 `real_puf_frontend_wrapper` 替换当前 `board_bringup_top` 中的固定/模拟响应源；
- 对真实板级采样得到的 raw response 做离线统计与常量重生成；
- 在不改后级主链（`response_aggregate_ctrl`、Hamming FE、SPONGENT/KDF、`auth_pass`）的前提下，重新验证认证闭环。

## 3. 当前起点
当前主链认证能够通过，使用的是：
- 注册冻结的受限挑战子表；
- 固定/模拟响应源；
- 固定版本的 `helper_xor`、`helper_mask`、`checksum`、`s_tag` 等常量。

因此，一旦前端换成真实采样数据，后级常量不能默认继续沿用，必须重新生成并重新绑定。

## 4. 输入数据
后续执行该流程时，至少需要三类输入：

### 4.1 原始采样数据
建议来自真实前端采样日志，例如：
- `board_id`
- `temperature_label`
- `session_id`
- `challenge_index`
- `challenge_hex`
- `repeat_index`
- `raw_resp_bit`
- `raw_resp_valid`

这些字段的详细口径见：
- `真实PUF前端采样数据字段与指标口径spec-2026-06-03.md`

### 4.2 注册参考数据
需要有“注册阶段”的参考响应材料，例如：
- 共同 challenge set 下的参考响应；
- 固定索引/受限挑战子表；
- 参考 `message_bits` 或根密钥材料。

### 4.3 当前后级参数约束
例如：
- `N_REP = 5`（默认）
- Hamming FE 参数
- SPONGENT/KDF 字段格式
- 当前协议消息口径

这些约束不应在本流程里随意更改，除非后续版本另行冻结。

## 5. 流程总览
建议按下面五步执行：

1. 真实采样数据清洗与分组；
2. `N_REP` 聚合，得到聚合响应；
3. 固定索引/受限挑战子表下的参考材料重建；
4. 重新生成认证所需常量；
5. 用新常量回灌当前主链并重新验证 `auth_pass`。

## 6. 详细步骤

### 6.1 数据清洗与分组
对真实采样日志按以下维度分组：
- 板号（`board_id`）
- challenge 编号（`challenge_index`）
- 温度/电压条件（如有）
- 重复采样序号（`repeat_index`）

需要先剔除：
- `raw_resp_valid = 0` 的无效样本；
- 明显缺失的 challenge；
- 非法重复次数。

### 6.2 聚合响应生成
在每块板、每个 challenge 上，按当前主链默认规则：
- `N_REP = 5`
- `response_aggregate_ctrl` 做多数表决（majority）
- 同时标出 `unreliable bit`

输出至少应包括：
- `aggregate_resp_bit`
- `unreliable_bit`
- `raw BER`
- `aggregate BER`

### 6.3 固定索引/受限挑战子表材料重建
根据注册阶段冻结的 challenge 子表和固定索引规则，生成与当前设备相匹配的响应材料。此步的目标不是单纯统计 PUF 指标，而是得到：
- 进入 Hamming FE 前的稳定比特材料；
- 可与注册参考响应对齐的 `message_bits` / `K` 恢复基础。

如果后续版本支持 per-device BestIdx / per-device challenge ROM，也应在此步完成设备级 challenge 子表的替换与绑定。

### 6.4 常量重生成
基于新的聚合响应与参考材料，重新生成：
- `helper_xor`
- `helper_mask`（若继续使用）
- `checksum`
- `s_tag`
- 必要时 `V_i`、`C_init` 或其它协议相关常量

这里要注意区分：
- 注册阶段固定的材料；
- 认证阶段会话相关的材料；
- 公开数据与保密数据。

建议将输出整理成明确的版本化文件，例如：
- `helper_xor_boardX_tempY.hex`
- `checksum_boardX_tempY.txt`
- `s_tag_boardX_tempY.txt`
- `binding_manifest_boardX_tempY.json`

### 6.5 回灌与认证重绑定
将新生成的常量回灌当前主链（wrapper 固定常量版本或后续动态加载版本），验证：
- `checksum_match`
- `auth_pass`
- `FE recovery success rate`
- `auth success rate`

如果 `checksum_match = 0` 或 `auth_pass = 0`，优先检查：
- challenge 子表是否一致；
- `helper_xor` 是否与当前板和当前温度条件匹配；
- `N_REP=5` 是否足够；
- 当前 FE 参数是否仍适合真实 BER。

## 7. 产物清单
后续执行该流程时，建议至少产出：

1. 原始采样日志
   - `raw_samples.csv`
2. 聚合汇总
   - `aggregate_summary.csv`
3. FE / 认证汇总
   - `fe_auth_summary.csv`
4. 常量绑定文件
   - `helper_xor`
   - `helper_mask`
   - `checksum`
   - `s_tag`
5. 版本说明文件
   - 记录 challenge 子表版本、板号、温度条件、采样时间、脚本版本

## 8. 验收指标
后续真实执行时，建议按三层验收：

### 8.1 原始 PUF 指标层
- raw BER
- reliability（稳定性）
- uniformity（均衡性）
- uniqueness（唯一性，仅限共同 challenge set）
- bit-aliasing（位偏置）

### 8.2 后处理/恢复层
- aggregate BER
- FE recovery success rate

### 8.3 认证链路层
- `checksum_match`
- `auth_pass`
- auth success rate

## 9. 当前不能直接得到的内容
在真实前端尚未上板采样前，下面这些内容不能提前写成已完成：
- 真实 `helper_xor` / `checksum` / `s_tag` 重生成结果；
- 真实板级 `auth_pass` 成功率；
- 真实 PUF 的 BER、reliability、uniformity、uniqueness、bit-aliasing；
- 真实温度/电压下的鲁棒性结论。

## 10. 与第一次 bring-up 的关系
第一次上板准备的重点是：
- 用固定/模拟响应源证明主链能跑通；
- 先完成 `start -> busy -> done -> auth_pass` 的板级闭环。

本流程属于第二阶段准备，重点是：
- 在真实前端接入后，用真实采样数据重建后级常量；
- 让认证闭环从“模拟响应源版本”过渡到“真实 PUF 版本”。

## 11. 简短结论
当前最合理的做法不是直接拿现有固定常量去套真实前端，而是先建立“真实采样 -> 聚合 -> 常量重生成 -> 认证重绑定”的标准流程。后续一旦拿到真实板级数据，就可以沿该流程逐步恢复 `auth_pass` 闭环，并统计真实 PUF 指标。
