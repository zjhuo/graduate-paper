## 二、已完成的验证

### 1. 标准 testbench
标准 testbench 已通过，说明当前主链在默认输入条件下可以正常完成一次认证流程。

通过条件：

- `auth_done=1`
- `checksum_match=1`
- `auth_pass=1`

### 2. 协议语义回归
协议语义回归 testbench 已通过，覆盖了：

1. 正确路径  
   - checksum 正确  
   - `s_tag` 正确  
   - 期望通过

2. 错误 checksum  
   - `checksum_match=0`
   - `auth_pass=0`

3. 错误 `s_tag`  
   - `checksum_match=1`
   - `auth_pass=0`

4. 错误 helper  
   - 不会误通过

5. 错误响应  
   - 不会误通过

当前结论：

- 正确路径能通过；
- 失败路径不会误通过；
- `auth_done` 表示流程结束，不等于认证成功；
- `auth_pass` 才表示当前认证通过。

### 3. bring-up 行为回归
板级 bring-up 风格 testbench 已通过，覆盖了：

1. 正常路径
2. `busy` 期间重复 `start`
3. 认证完成后再次 `start`
4. 中途 `reset` 打断后重新启动
5. 错误 checksum
6. 错误 `s_tag`

当前结论：

- 重复触发不会把流程弄坏；
- 完成后可以再次启动；
- 中途复位后可以恢复并重新启动；
- 错误 checksum / 错误 `s_tag` 不会误通过。

需要单独记下的一点：

- `recover_success_o` 在当前 bring-up 场景里不是稳定的板级成功标志；
- 后续如果要把它当正式板级状态口使用，需要单独冻结语义。

## 三、板级接口当前口径

### 正式板级口
当前建议保留为正式板级口的信号：

- `clk_i`
- `rst_ni`
- `start_i`
- `session_busy_o`
- `auth_done_o`
- `auth_pass_o`

### 可选状态口
当前可选保留：

- `recover_success_o`
- `checksum_match_o`

### 临时 bring-up / testbench 口
当前只适合作为临时口保留：

- `sample_req_o`
- `puf_resp_i`
- `puf_resp_valid_i`

当前口径下：

- `sample_req_o` 主要是 testbench / 外部响应源握手口；
- `puf_resp_i / puf_resp_valid_i` 主要是 testbench 或临时响应源输入；
- 它们当前不按正式 100MHz 板级高速同步接口处理。

### 不再保留的宽调试口
当前不建议恢复为正式板级口：

- `debug_key_o[31:0]`
- 其它宽 debug 总线

## 五、当前 timing 结论
前一阶段已经确认：

- synthesis timing 为正；
- 在当前“真实板级接口口径”下，post-route timing 也为正；
- 之前的 timing 问题主要来自 board-facing 输出口和约束方式，而不是 Hamming、SPONGENT 或 aggregate 核心逻辑。

当前处理结果是：

- `debug_key_o` 已移除；
- 状态口不再按 2ns 外部同步输出口去约束；
- `sample_req_o / puf_resp_i / puf_resp_valid_i` 当前按 bring-up / testbench 口径处理；
- timing 现在能够反映更真实的板级正式接口情况。

## 六、当前阶段还没有做的事
当前版本虽然已经可以作为上板前基线，但还没有进入这些内容：

1. 真实 PUF / 采样前端接入
2. 正式板级引脚分配与板卡工程打包
3. 板内模拟响应源的正式实现
4. PS / UART / AXI-Lite 方式的常量动态加载
5. 更完整的板级调试/观测手段（如 ILA）

## 七、当前建议的第一版上板思路
第一版上板优先采用：

- 板内固定/模拟响应源
- 不直接接真实 PUF/采样前端

当前目标不是一步到位做最终板级系统，而是先稳定验证：

- `start -> busy -> done -> auth_pass`

只要这条链在板上先跑通，后面再接真实前端时，问题范围会更可控。

## 八、当前结论
截至目前，可以把这版设计看成：

- 一版已经通过功能仿真；
- 已通过失败路径回归；
- 已通过 bring-up 行为回归；
- 已通过 Zynq-7010 方向 post-route 100MHz；
- 可以作为第一版上板前硬件主基线继续推进。

后续再往前走时，优先应该做：

1. 按既定口径准备第一版上板工程；
2. 固化最小 bring-up 流程；
3. 再决定何时接真实 PUF/采样前端。
