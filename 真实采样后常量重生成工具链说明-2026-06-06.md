# 真实采样后常量重生成工具链说明（2026-06-06）

## 1. 位置与目标

本文档说明 `D:\zijin\scripts\real_puf_binding\build_binding_from_samples.py` 在完整流程中的位置。

它的目标不是替代真实上板，也不是直接证明真实 PUF（物理不可克隆函数）认证已经成功，而是把后续 **“真实采样数据 -> 常量重生成 -> 认证重绑定”** 这条离线流程先搭成可执行骨架。

当前它服务的是第二阶段：

1. 真实前端采样得到 `raw_samples.csv` 或 `aggregate_summary.csv`；
2. 离线生成/整理 `helper_xor`、`helper_mask`、`checksum`、`s_tag` 等绑定材料；
3. 形成可回灌当前 wrapper（顶层封装/适配层）固定常量版本，或后续动态加载接口的文件；
4. 为真实前端接入后的重新验证做准备。

## 2. 与当前主链的关系

当前主链里：

- `response_aggregate_ctrl` 负责重复采样聚合；
- `hamming1611_core_stub` 负责 Hamming FE（模糊提取器）恢复；
- `spongent_core_stub` 负责 `checksum`、`K_hat`、`H_tag`、`S_tag`、`auth_pass` 相关后级计算；
- `board_bringup_top` 当前仍使用固定/模拟响应源和固定常量。

本工具链**不改**这些 RTL（寄存器传输级）模块，只是在真实采样数据到手后，准备后续需要回灌的材料文件。

## 3. 当前脚本

脚本位置：

- `D:\zijin\scripts\real_puf_binding\build_binding_from_samples.py`

它目前是一个**骨架版本**，主要完成：

- 从 `raw_samples.csv` 或 `aggregate_summary.csv` 读取单板数据；
- 构造聚合后的响应向量；
- 生成 Hamming(16,11) 代码偏移（code-offset）风格的 `helper_xor` 与 `helper_mask`；
- 按当前 `spongent_core_stub` 口径生成 `checksum`；
- 在提供占位会话字段时，按当前 `spongent_core_stub` 分段消息顺序生成 `s_tag`；
- 输出 `binding_manifest.json` 和 `fe_auth_summary.csv`。

## 5. 脚本参数与含义

### 5.1 必要参数

- `--raw-csv` 或 `--aggregate-csv`
  - 原始采样 CSV 或聚合汇总 CSV，至少提供一种。
- `--out-dir`
  - 输出目录。
- `--device-id`
  - 逻辑设备编号，例如 `device_001`。
- `--challenge-table-version`
  - 当前注册冻结的受限挑战子表 / challenge ROM 版本。
- `--salt-hex`
  - 当前 FE/KDF 使用的 salt。

### 5.2 常用参数

- `--board-id`
  - 当输入里有多块板时，指定当前要构建哪一块板的绑定材料。
- `--device-id-hex`
  - 若需要和当前硬件 `device_id_i` 对齐，可提供确切的 128 bit 十六进制值。
- `--target-bits`
  - 目标响应宽度，当前 Hamming(16,11) 骨架要求是 16 的整数倍。
- `--registration-rsel-hex`
  - 注册阶段参考响应向量（可选）。
- `--message-bits-hex`
  - 注册阶段消息位（可选）。
- `--message-bits-len`
  - `message_bits_hex` 的有效位宽（可选）。
- `--fill-missing`
  - 当观测 challenge 少于 `target_bits` 时，未观测位如何填充：
    - `registration`
    - `zero`

### 5.3 会话字段（当前可选）

下面这些字段**现在可以先占位**，后续真实流程里必须来自真实会话：

- `--nonce-d-hex`

- 脚本仍可生成 `helper_xor`、`helper_mask`、`checksum`；

## 7. 当前口径与边界

### 7.1 `helper_xor` / `helper_mask`

当前骨架按 **Hamming(16,11) 代码偏移** 方式生成：

- 先确定注册阶段的参考响应向量；
- 再确定消息位 `message_bits`；
- 通过 Hamming(16,11) 编码得到 codeword；
- `helper_xor = registration_rsel xor codeword`。

这一步是为了尽量贴近当前 `hamming1611_core_stub` 的接口形状，而不是声称已经等于真实最终 FE 方案全部细节。

### 7.2 `checksum`

当前脚本中的 `checksum` 尽量按 **当前 `spongent_core_stub` 的分段消息顺序** 生成：

- 前缀：`PUFv1 FE Hamming1611 checksum\n`
- `device_id`
- 换行
- `salt`
- `message_bits` 打包字节串

然后用当前 `spongent_ref.py` 的 `spongent_hash` 计算 128 bit 输出。

因此这里对齐的是：

- **当前 RTL / Python 参考里的主链口径**，
- 不是历史上其它 FE 参考脚本（如 repetition code / BCH code-offset）中的 checksum 口径。

### 7.3 `s_tag`

当前脚本中的 `s_tag` 只有在下面字段完整时才真正生成：

- `nonce_d`
- `nonce_s`
- `V_i`
- `C_init`

否则只会输出占位标记。

也就是说：

- 当前骨架支持 `s_tag` 的**格式和口径占位**；
- 但最终真实 `s_tag` 仍必须绑定真实会话字段和真实前端恢复出的 `K`。

## 9. 安全边界

需要明确：

- `helper` 可以公开，但不能被表述为“攻击者拿到 helper 就能推出长期密钥 K”；
- `checksum` 是绑定/完整性材料，不是会话密钥；
- `s_tag` 在最终协议里不应长期固定，它应由真实会话字段和服务器端/设备端一致密钥派生得到；
- 本工具链当前只是把“重生成和重绑定流程”骨架搭起来，不能被当成真实 PUF 安全性或真实认证成功的证据。

## 10. 当前最准确的定位

当前 `build_binding_from_samples.py` 的定位是：

- **离线工具链骨架**：已能从 synthetic 或未来真实采样数据生成绑定材料文件；
- **对齐当前主链口径**：尽量贴近 `hamming1611_core_stub + spongent_core_stub`；
- **不等于真实板级验证完成**：真实 `BER / reliability / auth success rate` 等仍需真实前端上板采样。
