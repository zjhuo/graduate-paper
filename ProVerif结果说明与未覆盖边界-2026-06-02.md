# ProVerif 结果说明与未覆盖边界

## 1. 文档目的

本文档用于解释当前第一版 ProVerif 最小模型：

- 已经验证了什么；
- 为什么当前这样抽象；
- 还没有覆盖哪些边界；
- 后续若继续增强模型，应从哪里往下补。

对应文件：

- `D:\zijin\ProVerif最小模型草案-2026-06-02.md`
- `D:\zijin\ProVerif最小模型-v1-2026-06-02.pv`

## 2. 当前模型的定位

当前 ProVerif 模型是**最小协议语义模型**，不是硬件实现细节模型。

它的目标不是证明：

- APUF 延迟是否合理；
- 5 次采样 majority 是否最优；
- Hamming 恢复是否正确；
- SPONGENT 轮函数实现是否无 bug；

而是先回答三个更核心的问题：

1. 会话密钥 `SK` 是否对攻击者保密；
2. 服务器接受设备时，是否确实存在对应设备发起过会话；
3. 设备接受服务器时，是否确实存在对应服务器参与过会话。

因此，这一版模型是：

> **协议安全语义的最小起点**

而不是最终完整形式化模型。

## 3. 当前抽象假设

### 3.1 helper 的处理

当前模型中：

- helper 按公开辅助数据处理；
- 攻击者可见 helper；
- 但攻击者不能只依赖 helper 推出长期密钥 `K`。

这与当前工程和论文口径一致：

- helper 是辅助恢复信息；
- 不是长期密钥本身；
- 第一版模型只保留这一安全边界，不展开 helper 内部结构。

### 3.2 FE 的处理

当前模型没有展开：

- APUF
- 逐次采样
- majority
- Hamming

而是把 FE 抽象为：

> **诚实设备能够本地恢复出与服务器一致的长期密钥 `K`**

这样做的原因是：

- 当前主要要验证的是协议层认证和会话密钥性质；
- 如果第一版就把 FE 细节全部建模，模型复杂度会明显上升；
- 但并不会立刻提升对这三条核心性质的可验证性。

### 3.3 SPONGENT/KDF 的处理

当前模型中：

- `hash(...)` 和 `kdf(...)` 只作为抽象函数出现；
- 不展开 SPONGENT 轮函数。

同时保留不同 domain separator：

- `dom_sk`
- `dom_h`
- `dom_s`

这能保证：

- `SK`
- `H_tag`
- `S_tag`

在模型里不会因为同一函数调用口径不清而混用。

## 4. 当前显式建模的协议消息

当前模型按四报文建模：

### M1

`(DeviceID, nonce_d)`

### M2

`(nonce_s, V_i, C_init)`

### M3

`(nonce_d, H_tag)`

### M4

`(S_tag)`

这与当前协议字段整理文档保持一致。

## 5. 当前显式建模的角色与事件

### 5.1 角色

当前模型显式建模了：

- `Device`
- `Server`

并显式使用：

- `nonce_d`
- `nonce_s`
- `H_tag`
- `S_tag`
- `SK`

### 5.2 事件

当前模型定义了四个关键事件：

- `DeviceStart`
- `ServerStart`
- `ServerAcceptDevice`
- `DeviceAcceptServer`

这些事件的用途是：

- 为双向认证查询提供“谁在什么条件下接受了对方”的形式化锚点。

## 6. 当前已跑出的结果

在本机 ProVerif 2.05 环境下，当前 `.pv` 模型已经实际运行，并得到：

### 6.1 会话密钥保密性

当前查询结果为：

- `SK secrecy`：**true**

也就是：

> 当前最小模型下，攻击者不能推出模型中的会话密钥派生结果。

### 6.2 服务器认证设备

当前查询结果为：

- `ServerAcceptDevice ==> DeviceStart`：**true**

也就是：

> 如果服务器接受了某次设备认证，那么一定存在一个对应设备发起过该轮会话。

### 6.3 设备认证服务器

当前查询结果为：

- `DeviceAcceptServer ==> ServerStart`：**true**

也就是：

> 如果设备接受了某次服务器认证，那么一定存在一个对应服务器参与了该轮会话。

## 7. 当前结果应该怎样理解

当前结果说明的是：

- 在当前抽象前提下；
- 在当前四报文语义下；
- 在 `K` 作为 Device / Server 共享私有长期密钥的前提下；

这套最小协议模型满足：

1. 会话密钥保密；
2. 服务器不会无中生有地接受一个并未启动的设备会话；
3. 设备不会无中生有地接受一个并未参与的服务器会话。

但当前结果**不等于**：

- 真实硬件一定安全；
- helper 任意构造都安全；
- 真实板级采样误差不会影响系统；
- 所有状态推进与异常处理都已被形式化证明。

## 8. 当前模型没有覆盖的内容

下面这些内容目前**没有进入第一版 ProVerif 模型**。

### 8.1 真实硬件前端行为

未覆盖：

- APUF 延迟
- challenge settle cycles
- raw BER
- 5 次采样 / 7 次采样切换
- `response_aggregate_ctrl` 的具体 majority 过程

原因：

- 这些更偏硬件/统计行为；
- 当前最小模型先只关心协议认证语义。

### 8.2 Hamming / FE 解码细节

未覆盖：

- helper_mask
- helper_xor 的具体比特结构
- Hamming 解码步骤
- 解码失败路径

原因：

- 当前只保留“诚实设备可恢复 K”这一抽象；
- 不在第一版里建模 FE 内部电路/算法。

### 8.3 SPONGENT 轮函数细节

未覆盖：

- SPONGENT 吸收/挤出内部步骤
- 轮函数
- digest 输出寄存行为

原因：

- 第一版只需要 `hash/kdf` 抽象；
- 无需把实现级结构带进协议验证。

### 8.4 状态推进与异常处理

未覆盖：

- `Seed_i -> Seed_next`
- 失败不更新状态
- 去同步恢复
- 缓存超时
- 重试策略

这部分当前只保留为文字安全性说明，不进入第一版 ProVerif。

## 9. 当前模型的边界与合理性

### 9.1 为什么当前边界是合理的

当前这样抽象的好处是：

1. 先把协议的核心认证语义站住；
2. 避免一开始就把硬件统计细节和协议消息语义混在一起；
3. 让最小模型更容易跑通并解释；
4. 给后续增强模型留下明确扩展方向。

### 9.2 当前边界的局限

当前模型的局限也很明确：

- 它没有回答真实 PUF 前端的工程问题；
- 没有回答 helper 重生成或 BER 变化时的安全边界；
- 没有形式化证明失败会话不更新状态；
- 没有建模多轮链式状态推进。

因此：

> **当前模型适合作为“第一版协议安全语义证明”，不应被误解成最终完整安全证明。**

## 10. 后续增强方向

如果下一步继续增强 ProVerif 线，建议按下面顺序推进。

### 第一优先：把状态推进补进模型

补：

- `Seed_i`
- `Seed_next`
- 认证成功后状态推进

这样可以更贴近当前协议文档。

### 第二优先：补失败不更新状态

把当前文档口径：

- 失败不更新状态

显式转成模型假设或过程逻辑。

### 第三优先：增强 secrecy 查询

当前 `SK secrecy` 已经不是固定常量级别的弱查询，  
但后续还可以进一步加强成：

- 与接受事件更紧密绑定的 secrecy 说明；
- 或在更完整状态模型中表达“被接受会话的 SK 不泄露”。

### 第四优先：再考虑去同步/缓存/重试

这些更适合放到第二阶段形式化工作里，而不是塞进第一版最小模型。

## 11. 当前建议结论

当前最合理的结论是：

1. 第一版 ProVerif 最小模型已经可运行；
2. 三条核心目标已经得到正结果：
   - `SK secrecy`
   - Server authenticates Device
   - Device authenticates Server
3. 当前模型只证明协议最小语义，不覆盖硬件前端和状态异常处理全部细节；
4. 若后续继续推进，优先补“状态推进”和“失败不更新状态”，而不是先去扩硬件细节。
