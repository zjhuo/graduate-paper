# APUF前端接入实验结果与边界说明（2026-06-04）

## 1. 目的
本文档用于收口当前两条 APUF 前端实验分支的结论，明确“哪些内容已经验证”“哪些内容尚未验证”，避免将“前端接线和时序骨架已通”误写成“真实 APUF 响应已经通过认证”。

## 2. 当前实验分支
当前在 `D:\zijin\rtl` 下新增并验证了两条实验分支：

1. 结构化 APUF RTL 前端分支
   - `real_puf_frontend_wrapper.sv`
   - `iotpufs_terminal_board_real_puf_top.sv`
   - `iotpufs_terminal_board_real_puf_top_tb.sv`
   - 前端核心：`D:\zijin\hardware\apuf\rtl\apuf64_fpga.v`

2. 行为仿真 APUF 前端分支
   - `sim_puf_frontend_wrapper.sv`
   - `iotpufs_terminal_board_apuf_sim_top.sv`
   - `iotpufs_terminal_board_apuf_sim_top_tb.sv`
   - 前端核心：`D:\zijin\hardware\apuf\rtl\apuf64_sim.v`

两条分支都保持了后级主链不变，即：
- `response_aggregate_ctrl`
- Hamming FE
- SPONGENT/KDF
- `protocol_fsm_stub`
- `auth_pass`

实验目标不是直接得到最终可用的真实 PUF 认证结果，而是先验证：
- 前端 wrapper 接口是否合理；
- `sample_req -> challenge -> response -> valid` 这条链是否能接入现有主链；
- 现有主链在接入新前端后是否仍能跑到 `auth_done`。

## 3. 已验证结果

### 3.1 wrapper 控制骨架已通
`real_puf_frontend_wrapper_tb.sv` 已验证：
- 只有在 `sample_req_i && challenge_valid_i` 时才启动；
- 能完成 challenge 锁存、等待 settle、产生 `launch`、等待采样、输出 `puf_resp_valid_o`；
- busy 期间不会重复接单；
- timeout 路径可以工作。

这说明前端 wrapper 的接口和最小时序骨架是成立的。

### 3.2 两条实验分支都能跑到 `auth_done`
当前两条前端实验分支的仿真结果都能走到：
- `auth_done = 1`

这说明：
- 前端 wrapper 与 `iotpufs_terminal_top` 的连接关系是通的；
- `response_aggregate -> Hamming FE -> SPONGENT/KDF -> protocol FSM` 这条后级链没有因为前端接法变化而失效。

## 4. 当前未通过的结果
两条实验分支目前都没有得到认证通过结果：
- `checksum_match = 0`
- `auth_pass = 0`

### 4.1 结构化 APUF RTL 分支
`iotpufs_terminal_board_real_puf_top_tb.sv` 的最小输出为：
- `auth_done = 1`
- `checksum_match = 0`
- `auth_pass = 0`

诊断中间量显示：
- `rsel_vector` 近似全 1；
- `message_bits_hat` 近似全 1。

这说明 `apuf64_fpga.v` 在当前纯数字仿真环境下，没有体现出真实物理 APUF 的 challenge 敏感性，而是退化为近似常值响应源。

### 4.2 行为仿真 APUF 分支
`iotpufs_terminal_board_apuf_sim_top_tb.sv` 的最小输出为：
- `auth_done = 1`
- `checksum_match = 0`
- `auth_pass = 0`

诊断中间量显示：
- `rsel_vector` 近似全 0；
- `message_bits_hat` 近似全 0。

这说明当前 `apuf64_sim.v` 在现有配置下，虽然比 `apuf64_fpga.v` 更像“仿真模型”，但在当前这条最小接入实验中仍然没有形成可直接继承现有固定常量闭环的响应材料。

## 5. 为什么当前 `auth_pass` 不通过不意外
当前固定常量（如 `helper_xor`、`checksum`、`s_tag` 等）是围绕原先的固定/模拟响应源闭环生成的。当前前端实验分支换成：
- `real_puf_frontend_wrapper + apuf64_fpga`，或
- `sim_puf_frontend_wrapper + apuf64_sim`

之后，前端响应材料已经改变，因此不能期待后级仍直接复用原来的固定常量而继续认证通过。

此外，需要特别强调：
- `apuf64_fpga.v` 的当前纯数字仿真不等价于真实 FPGA 物理延迟竞争；
- `apuf64_sim.v` 的当前行为模型也不等于真实板级采样结果。

因此，当前实验分支失败，不能解读为“主链错误”，更不能解读为“真实 APUF 已验证失败”。

## 6. 当前已经证明了什么
当前实验已经证明：
1. `real_puf_frontend_wrapper` 的接口定义和最小时序骨架是可行的；
2. 将前端 wrapper 接入当前主链的连接关系是成立的；
3. 后级 `response_aggregate`、Hamming FE、SPONGENT/KDF、`auth_pass` 链路在接入新前端后仍能走到 `auth_done`；
4. “前端 wrapper 替换当前模拟响应源”的总体工程路径是可实施的。

## 7. 当前还没有证明什么
当前实验**没有证明**：
1. 真实 APUF 响应已经驱动通过 `auth_pass`；
2. `apuf64_fpga.v` 的纯数字仿真足以代表真实物理 APUF 前端；
3. 当前固定常量可以直接迁移到真实前端版本；
4. 真实 PUF 指标（BER、reliability、uniformity、uniqueness、bit-aliasing）已经得到。

## 8. 当前最准确的阶段结论
当前应写成：
- 基础 64 级 APUF RTL 已存在；
- 真实前端 wrapper 的接口和时序骨架已经验证通过；
- 前端 wrapper 已能接入现有认证主链并跑到 `auth_done`；
- 但当前仍未完成真实 APUF 前端验证，也未完成基于真实 APUF response 的 `auth_pass` 闭环验证。

不应写成：
- 真实 APUF response 已经在当前硬件主链中认证通过；
- 真实 PUF 指标已经通过当前前端实验得到。

## 9. 后续最小接入路径
后续最小接入路径仍然是：
1. 保持 `response_aggregate_ctrl`、Hamming FE、SPONGENT/KDF、`auth_pass` 后级主链不变；
2. 用 `real_puf_frontend_wrapper` 替换 `board_bringup_top` 中的固定/模拟响应源；
3. 在真实板级条件下，通过采样确定：
   - challenge settle cycles；
   - launch pulse width；
   - sample delay；
   - timeout cycles；
4. 基于真实前端采样结果重新生成：
   - `helper_xor`
   - `checksum`
   - `s_tag`
   - 以及相关参考材料；
5. 再验证真实前端条件下的：
   - `aggregate BER`
   - FE recovery success rate
   - `auth_pass`

## 10. 简短结论
当前这两条前端实验分支的主要价值是：
- 已证明“真实前端 wrapper 能接进去”；
- 已证明“后级主链不会因为前端替换而立刻失效”；
- 但还没有证明“真实 APUF response 已经跑通认证”。

因此，当前最稳妥的结论是：
> 前端接法和时序骨架已验证，真实 PUF 有效性与真实认证成功仍需依赖后续板级采样和重新绑定常量来确认。
